# AI 作業範囲ガード（Scope Guard）

> プロジェクト横断の普遍規約。Claude Code の「指示範囲外の自走」を防ぐ。

---

## 共通原則

依頼達成に**必要**な従属変更（型・シグネチャ更新、依存テスト追従、共通関数の引数変更に伴う呼び出し元修正等）は**許容**する。
それ以外の変更は以下 5 項目で**禁止**する。

「自分が触れたついでに直したくなった」は依頼達成に必要ではない。別タスクに切り出す。

---

## 共通例外テンプレート

禁止項目に該当する変更を行う場合は、以下 **4 条件をすべて満たす**ときのみ例外として許可される:

1. **依頼達成に必須である**（外すと依頼が完了しない）
2. **変更理由を 1 行で説明できる**（「なんとなく」「念のため」は不可）
3. **影響範囲（変更ファイル・関数）を列挙できる**
4. **根拠ある検証がある**（テスト・型チェック・実行確認のいずれか）

レビュー時はこの 4 条件で逸脱の可否を判断する。

---

## 禁止項目

### 1. 指示範囲外のリネーム
依頼で言及されていないシンボル（変数・関数・クラス・ファイル名）の rename を行わない。
リネームは git blame を分断し、レビュー負荷を増やし、merge conflict を生む。

### 2. 依頼外の広範囲・意味非変更 diff
import 並び替え、インデント正規化、空行整理、コメント整形等の**意味を変えない大量 diff** を依頼外で行わない。
**対象外**: 保存時の自動整形（editor format-on-save）、局所的な linter --fix（同一関数内の警告解消等）。

### 3. 依頼達成に不要な改修（ついでリファクタ）
修正対象外ファイルへの「ついでリファクタ」を行わない。
共通原則で許容される**従属変更**（依頼の達成に必要な波及修正）とは厳密に区別する。「触れたから直す」は不可、「直さないと依頼が完了しない」のみ可。

### 4. テスト期待値の根拠なき書き換え
テスト失敗を「期待値が古い」で済ませない。
書き換えるなら**「なぜ前の期待値が誤りか」を 1 行で根拠付ける**。期待値の方が正しくコードがバグの可能性を常に検討する。

### 5. 既存実装の巻き戻し
自分が読み解けない既存ロジックを「不要」と判断して削除・簡略化しない。
理由が読み解けないコードには通常**過去の事故対応・制約・歴史的経緯**が埋め込まれている。`git log -p <file>` で背景を確認するか、ユーザーに尋ねる。

---

## タグ命名規約

逸脱を検知した場合（自分の変更でも他者のレビューでも）、レッスン記録時に **[scope-creep]** タグで記録する。

タグの使い分け:
- `[scope-creep]` — 上記 5 項目のいずれかに該当する逸脱
- `[no-test-mutation]` — 特に項目 4（テスト期待値の根拠なき書き換え）に特化した逸脱

記録の運用フロー（いつ・誰が・どの経路で記録するか）は本ガード文書ではなく、プロジェクトのレッスン記録運用に従う。本文書はタグの命名規約のみ提供する。
