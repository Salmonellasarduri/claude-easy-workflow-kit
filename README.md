# claude-easy-workflow-kit

Claude Code を、**「次に何をするか迷わず進める」ためのワークフローキット**です。

このキットを入れると、開発を 7 つの `/command` で順番に進められるようになります。
やることは難しくありません。**コマンドを打って、出てきた選択肢を選ぶだけ**です。

- 何から始めればいいか分からない
- 実装は進むけど、テストやレビューが抜けやすい
- 作業を中断すると、再開が面倒
- 記録やログが毎回ばらつく

こうした「会話ベース開発で起きやすい詰まりどころ」を減らして、
**考える → 作る → 確かめる → 残す** を自然に回せるようにします。

プログラミングに詳しくなくても使えます。
レビューや整理は Claude が進め、必要なら Codex などの外部ツールも補助に使えます。
ただし、**外部ツールはなくても動きます**。

---

## これは何か

`claude-easy-workflow-kit` は、Claude Code 用のワークフロー定義セットです。

含まれているもの:

- `commands/` — 7つのワークフローコマンド
  - `strategy.md` / `plan.md` / `implement.md` / `debug.md` / `review.md` / `save.md` / `restart.md`
- `rules/` — ワークフローのルールとガイド
  - `workflow-gate.md` / `research-guide.md` / `doc-update-guide.md`
- `schemas/` — コマンド間の受け渡し形式
  - `handoff.md`
- `workflow.yaml` — パス・ツール・振る舞いの設定
- `scaffold.sh` / `scaffold.ps1` — セットアップスクリプト

単なるコマンド集ではなく、**開発の流れそのものを整えるキット**です。

---

## こんな人に向いています

- Claude Code を使っている（使い始めたばかりでもOK）
- 初見でもすぐ使える形がほしい
- 実装だけでなく、確認やレビューも抜けなくしたい
- 個人開発や少人数開発で進め方をそろえたい
- AI に作業を進めてもらいつつ、自分は判断に集中したい

逆に、Claude Code を使わない環境には向いていません。

---

## まず最初にやること

導入したら、まずは Claude Code でこれを入力してください。

```text
/strategy
```

やりたいことが決まっているなら、一言足すだけで使えます。

```text
ログイン機能を追加したい /strategy
```

すると、Claude Code がプロジェクトを読みながら、

- 何を作るか
- どこから着手するか
- 先に確認すべきことは何か
- 次にどのコマンドへ進むべきか

を整理してくれます。

**迷ったら `/strategy` から始める**。
これだけ覚えておけば使い始められます。

## Quick Start

### Unix / macOS / WSL

```bash
git clone https://github.com/Salmonellasarduri/claude-easy-workflow-kit.git
cd claude-easy-workflow-kit
./scaffold.sh /path/to/your/project
```

### Windows PowerShell

```powershell
git clone https://github.com/Salmonellasarduri/claude-easy-workflow-kit.git
cd claude-easy-workflow-kit
.\scaffold.ps1 -ProjectPath C:\path\to\your\project
```

### 手動で入れる場合

`.claude/` フォルダに以下を配置しても使えます。

```text
commands/*.md  → .claude/commands/
rules/*.md     → .claude/rules/
schemas/       → .claude/schemas/
workflow.yaml  → .claude/workflow.yaml
```

## 7つのコマンド

### `/strategy`
**何をどう進めるかを整理する入口**です。

仕様がふわっとしているときや、最初の一歩が分からないときに使います。
Claude Code がコードベースや状況を見て、進め方の候補を出します。

使いどころ:

- 新機能を入れたいが、どこから触るべきか分からない
- 変更範囲が大きそうで、先に全体像を見たい
- まず調査だけしたい

例:

```text
/strategy
```

```text
管理画面に検索機能を追加したい /strategy
```

---

### `/plan`
**実装に入る前の作業計画を作る**コマンドです。

変更対象、進める順番、テスト観点、リスクなどを整理します。
大きめの変更や、あとから見返したい作業に向いています。

使いどころ:

- 手戻りを減らしたい
- いきなり実装すると不安
- 複数ファイルにまたがる変更を整理したい

例:

```text
/plan
```

```text
プロフィール編集機能の作業計画を作って /plan
```

---

### `/implement`
**実際の実装を進める**コマンドです。

計画がある場合はそれに沿って進め、ない場合でも今の文脈から必要な変更を進めます。
小さな修正なら、ここから始めても問題ありません。

使いどころ:

- 追加・修正したい内容がはっきりしている
- まず動く形を作りたい
- 軽微な UI 修正やバグ修正を素早く進めたい

例:

```text
/implement
```

```text
フォーム送信後に成功メッセージを出すようにして /implement
```

---

### `/debug`
**動作確認や不具合の切り分け**を進めるコマンドです。

テスト実行、再現手順の整理、ログ確認、原因候補の洗い出しなどに使います。
「動かない」「たまに壊れる」「何が悪いか分からない」に強いコマンドです。

使いどころ:

- エラーの原因を探したい
- テストが落ちた理由を見たい
- 修正後の確認をしたい

例:

```text
/debug
```

```text
ログイン後に画面遷移しない原因を調べて /debug
```

---

### `/review`
**コードレビューを通して、抜けやリスクを見つける**コマンドです。

実装した内容を見直し、バグの可能性、考慮漏れ、テスト不足、設計上の違和感などを洗い出します。
レビューは Claude が進めます。外部ツールがあれば追加レビューもできます。

使いどころ:

- 自分で書いた変更を客観的に見直したい
- コミット前に安全確認したい
- 「動くけど、このままでいいか不安」を解消したい

例:

```text
/review
```

```text
今回の変更をレビューして、問題があれば優先度順に出して /review
```

---

### `/save`
**コミットや記録をまとめる**コマンドです。

変更内容の要約、コミットメッセージ案、作業ログの整理などを行います。
「実装したまま終わる」を防いで、あとから追いやすい状態にします。

使いどころ:

- 作業を一区切りつけたい
- 何を変更したかを残したい
- コミット文や記録を毎回考えるのが面倒

例:

```text
/save
```

```text
今回の変更を保存用に整理して /save
```

---

### `/restart`
**中断した作業を再開しやすくする**コマンドです。

タスクの状態、途中までの内容、次にやることを読み直して、再開の足場を作ります。
日をまたいだ作業や、複数のタスクを行き来するときに便利です。

使いどころ:

- 昨日の続きがすぐ思い出せない
- 途中で止まった作業に戻りたい
- いま何が終わっていて、何が残っているか確認したい

例:

```text
/restart
```

```text
途中だった通知機能の作業を再開したい /restart
```

## おすすめの使い方

### 標準の流れ

最初はこの順番で使うのがおすすめです。

1. `/strategy`
2. `/plan`
3. `/implement`
4. `/debug`
5. `/review`
6. `/save`

この流れにしておくと、
**考え漏れ・確認漏れ・記録漏れ**がかなり減ります。

### 小さな修正なら短縮版でもOK

変更内容が明確で小さい場合は、次の流れでも十分です。

```text
/implement → /debug → /review → /save
```

たとえば、

- 文言修正
- 軽微なバグ修正
- 小さな UI 調整
- 既存機能への単純な追加

のようなケースです。

## 導入後に作られるもの

セットアップすると、作業を回しやすくするためのファイル群が用意されます。
既にあるファイルは基本的にそのまま使い、足りないものを補う前提です。

| ファイル | 用途 |
|---------|------|
| `tasks/current.md` | 今やっている作業の整理 |
| `tasks/lessons.md` | 次回以降に活かす反省・学びの記録 |
| `docs/ROADMAP.md` | 全体の進み具合や今後の見通し |
| `docs/DEVLOG.md` | 開発ログの入口 |
| `docs/devlog/` | セッションごとの詳細ログ |

「その場では分かっていたけど、後で分からなくなる」を防ぐための土台です。

---

## `workflow.yaml` について

導入すると `.claude/workflow.yaml` が置かれます。
**基本的には触らなくて大丈夫です。**

このファイルには、たとえば次のような設定が入ります。

- どのファイルをタスク管理に使うか
- どこにログやロードマップを置くか
- 外部ツールを使うかどうか
- ワークフロー上の振る舞いをどうするか

既存プロジェクトの構成に合わせたいときだけ調整してください。

例:

```yaml
paths:
  tasks: my-project/tasks.md
  roadmap: CHANGELOG.md
```

デフォルトのままでも使えるようにしてあるので、
最初は「設定ファイルがある」くらいの理解で問題ありません。

## 外部ツール連携

このキットは**単体で動作します**。
以下の外部ツールは、入っていればより便利になるオプションです。

| ツール | できること | 必須？ |
|--------|------------|--------|
| [Codex CLI](https://github.com/openai/codex) | レビューや計画確認の補助 | いいえ |
| [Gemini CLI](https://github.com/google-gemini/gemini-cli) | 広めの調査や情報整理の補助 | いいえ |
| [GitHub CLI](https://cli.github.com/) | Issue や Milestone 操作の補助 | いいえ |

必要なら `.claude/workflow.yaml` で有効化できます。

```yaml
tools:
  codex: true
  gemini: true
  gh: true
```

ポイントは、**なくても使えること**です。
まずは Claude Code だけで始めて、必要になったら足してください。

## このキットの考え方

Claude Code は強力ですが、自由度が高いぶん、

- 毎回やり方がぶれる
- 実装までは行くが、その後が雑になる
- 中断からの再開コストが高い
- 小さな学びが蓄積されにくい

ということが起きやすくなります。

このキットは、それを防ぐために作られています。
目指しているのは、AI に全部丸投げすることではなく、
**人は判断に集中し、手順はできるだけ選ぶだけにする**ことです。

ベースになっているのは、
[Artificial Personality](https://github.com/Salmonellasarduri/Artificial-Personality) での 70 回以上の開発セッションです。
実際に回して分かった「詰まりやすい点」を、日常的に使いやすい形にまとめています。
