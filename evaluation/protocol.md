# 評価プロトコル: CEWK 構造化ウォークスルー

> claude-easy-workflow-kit (CEWK) vs superpowers vs 素 Claude Code
> 「文系ユーザーが空ディレクトリから迷わず使えるか」の実運用 UX 比較

---

## 評価の範囲

**この検証は「空ディレクトリ起点の実運用 UX 比較」であり、ツールの総合的な優劣比較ではない。**
結果の主張には「今回の条件・このタスクでは」の限定を付ける。

## 実施順序

学習効果バイアスを最小化するため、CEWK を最後（最も不利な条件）にする:

```
1. superpowers  → 2. bare（素 Claude Code） → 3. CEWK
```

各環境は **別セッション** で実行する（前セッションのコンテキストを持ち込まない）。

---

## 評価の2段階

### Stage 0: 導入 UX
空ディレクトリから「キットが使える状態」になるまで。

| 環境 | 導入手順 |
|------|---------|
| bare | 何もしない（空ディレクトリで Claude Code を起動） |
| superpowers | `/plugin install superpowers@claude-plugins-official` → 再起動 → `/using-superpowers` |
| CEWK | `scaffold.ps1 -ProjectPath <dir>` を実行 |

記録: 操作数、所要時間、詰まった箇所、知識要求度、技術用語数、次アクション明確さ

### Stage 1: タスク遂行 UX
導入完了後、同一タスクを実行して完了するまで。

---

## タスク定義

**開始プロンプト（3環境で同一）:**

```
自己紹介ページを作ってブラウザで確認できる状態にしてほしい。
名前は「田中太郎」、趣味は読書と散歩。シンプルなデザインで。
```

**完了条件:**
ブラウザで自己紹介ページが表示され、名前・趣味が正しく表示されている。

**Phase A（主評価）:** ローカル完成まで
**Phase B（補足）:** 公開まで（今回はスキップ）

---

## 異常系シナリオ（3種、Phase A 完了後に実行）

### 異常系 1: 技術的異常
Phase A 完了後、index.html の `</div>` を1つ手動で削除し、以下を伝える:
```
ページの表示が崩れちゃった。直してほしい。
```

### 異常系 2: 期待値ズレ
```
もっとシンプルにして。色も暗めに変えて。文字は大きめがいい。
```

### 異常系 3: 環境異常
Git 未初期化（`git init` していない）の状態で:
```
これを保存して。
```

---

## セッション再開テスト

1. Phase A + 異常系が終わったらセッションを終了する
2. **新しい Claude Code セッション**を同じディレクトリで起動する
3. 以下を伝える:

```
さっき作った自己紹介ページの色を変えたい。もうちょっと明るい感じにして。
```

**成功条件:**
- 前回の成果物を特定できる
- 変更対象を提案できる
- ユーザー確認後に修正できる
- ブラウザ確認まで完了する
- **5ターン以内**で上記すべてを達成

---

## 記録方法

- 各判断ポイントは `log-template.md` に記入
- **moat 観測項目**（rubric.md 参照）は各判断ポイントおよび異常系シナリオで記録する:
  - 技術用語数: Claude の出力テキストをカウント
  - 次アクション明確さ: 各判断ポイントで3値（明確/推測可能/不明）を判定
  - プログレス可視性: セッション全体を通して、進捗表示が1回でもあれば「有」
- 代表的な画面は **スクリーンショット** を撮り、`evaluation/screenshots/` に保存
- ファイル名: `{環境}_{stage}_{連番}.png`（例: `cewk_stage1_03.png`）
- 全セッションの **タイムスタンプ** を記録（開始時刻、TTTV、各異常系の回復時間）
