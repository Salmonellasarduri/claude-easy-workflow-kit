# 採点ルーブリック

---

## 技術知識要求度（3段階）

| レベル | 定義 | 判定基準 | 例 |
|--------|------|---------|-----|
| **不要** | 日常語だけで回答でき、どの選択肢を選んでも安全 | 技術用語ゼロ **かつ** 不可逆性なし **かつ** 選択肢間の結果差分が小さい | 「どんな雰囲気がいいですか？」 |
| **軽い知識** | 技術用語が出るが文脈から推測可能、または結果差分が小さく間違えても修正容易 | 技術用語1つ以上 **だが** 文脈で意味が推測できる **かつ** 不可逆性なし | 「フォルダ名をどうしますか？」 |
| **専門知識** | 技術用語の事前知識が必要 **または** 選択肢間の結果差分を理解しないと選べない不可逆選択 | 以下のいずれか: (a) 技術用語の意味を知らないと判断不能 (b) 不可逆で結果差分が大きい選択 | 「public リポジトリにしますか？」 |

---

## 追加フラグ

### 不可逆選択フラグ
- **Yes**: 取り消しが効かない、または取り消しに専門知識が必要な判断
- **No**: やり直し可能

## 判断ポイントの切り出し基準

### 定義
**判断ポイント = ユーザーが次の行動を選ぶ必要がある1つの瞬間**

### ルール
1. Claude が質問し、ユーザーが回答する → **判断ポイント**
2. Claude が自動で進めた部分（ユーザーの入力不要） → **数えない**
3. Claude が進捗報告だけして「続けますか？」→ **判断ポイント**（ただし知識要求度は「不要」）

### ターン数 vs 内包判断数
- **ターン数**: ユーザーが入力した回数
- **内包判断数**: 1ターンに含まれる独立した判断の数

例:
| Claude の発言 | ターン数 | 内包判断数 |
|---|---|---|
| 「どんな色がいいですか？」 | 1 | 1 |
| 「サイト名と配色を教えてください」 | 1 | 2 |
| 「A: React / B: HTML / C: Vue どれにしますか？」 | 1 | 1 |

## moat 観測項目

| 項目 | 計測方法 |
|------|---------|
| **選択肢の技術用語数** | 各判断ポイントで Claude が提示したテキスト中の技術用語をカウント |
| **エラー説明の技術用語数** | 異常系シナリオで Claude が返した説明文中の技術用語をカウント |
| **次アクション明確さ** | 3値: 明確（一意に分かる）/ 推測可能（複数候補だが推測できる）/ 不明 |
| **プログレス可視性** | 今どの段階にいるかが明示されるか: 有 / 無 |

### 技術用語の定義
以下に該当する語を「技術用語」としてカウントする:
- プログラミング言語名・フレームワーク名（HTML, CSS, React, npm 等）
- 開発ツール名（Git, GitHub, terminal 等）
- 開発概念（リポジトリ, ブランチ, デプロイ, ビルド, コンパイル 等）
- ファイルシステム用語（パス, ディレクトリ, 拡張子 等）— ただし「フォルダ」「ファイル」は日常語として除外

---

## 採点の注意事項

- Claude の**実際の出力テキスト**に基づいて判定する（設計意図ではなく実測）
- 判定に迷った場合は「専門知識」側に倒す（保守的に採点）
- 判定根拠を log の「メモ」欄に記載する
