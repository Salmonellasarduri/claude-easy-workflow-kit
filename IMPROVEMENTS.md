## P0: プロダクト設計の根本改善

### 選択肢の非技術化（松竹梅フレーミング）
- `/strategy` が提示する A/B/C を技術用語で書かない
- 「コスト優先 / 品質優先 / スピード優先」のように、文系ユーザーが直感で選べる軸にする
- strategy.md のプロンプトに「技術用語を排除した選択肢を提示せよ」を明記

### エラー時の「やさしい説明」フェイルセーフ
- 連鎖が途中で壊れたとき、AIが「何が起きたか」を文系言語で説明する
- 自動で `/debug` に誘導する設計
- 「エラーが出てもAIがなんとかしてくれる」という確信を作る

### メッセージングの矛盾修正
- 「全自動」と「4回止まる」は矛盾 → 正確には「半自動」
- README/コマンド説明を「ガイド付き半自動フロー」に統一
- 期待値を正しくセットする

## P1: UX 改善

### 「7コマンド」表現の見直し
- ユーザーが打つのは `/strategy` だけ → 「1コマンドで始まる4ステップのガイド」
- 内部の7フェーズはユーザーに見せる情報ではない
- README の「7つのコマンド詳細」セクションを折りたたみ or 末尾に移動

### オプション外部ツールの段階的開示
- Codex/Gemini/GH CLI の存在が「自分のセットアップは最適でないのでは」という不安を生む
- 初回は完全に隠す → 慣れてきたらアンロックする設計
- または README で「まず使ってみて、物足りなくなったら」の位置に移動（現状もこれに近い）

### /restart の拡張 → 部分的巻き戻し
- 現在は「どこまで進んだかを検出して続行」
- 線形フロー以外（バグ修正、差し戻し）で部分的に戻れるように
- 「全部やり直し」ではなく「このステップからやり直し」

## P2: コンテンツ・マーケティング

### 70+ セッションのケーススタディ化
- 実戦データからケーススタディ集を作る
- 「この人はこういうタスクをこう進めた」が最強のオンボーディング素材
- 文系ユーザーは抽象ドキュメントより具体事例から学ぶ

### 判断ポイントの「教育機会」再フレーム
- 4回の判断 = 「AIとの協働を学ぶ4つの練習」
- 「今あなたは設計フェーズを体験しています」のようなナラティブ
- ツール + 教材というポジショニング

### 日本語コミュニティ先行者利益
- Zenn / note / Qiita での情報発信
- 日本語での Claude Code 活用情報はまだ薄い
- 英語圏の競合が手を出しにくいニッチ

## P3: 将来検討

- コマンド恐怖層 / 英語README離脱層 / 環境構築外注層

- Claude 単体ではコンテキスト忘却が起きる
- /save で構造化保持する CEWK は中長期プロジェクトで ROI が明確

## Round 2 追加 (2026-04-02)

### P0 追加: プログレス表示（「今何合目か」）
- 連鎖中に「今、裏で何が起きているか分からない」心理的不安
- 各ステップ終了時に「今 3/6 合目です」のように日本語で進捗を表示
- superpowers（英語・開発者向け）との決定的な差になりうる

### P1 追加: 異常系 = 「期待値のズレ」対応
- 文系ユーザーの異常系は「ビルドエラー」ではなく「思ったのと違う（ダサい、動かない）」
- 「なんか違う」を言うだけで修正ループに入れる設計
- superpowers(TDD)は技術的異常に強いが、期待値ズレにはCEWKが柔軟に対応できるはず

### P2 追加: lessons.md のB2B資産化
- 「文系ユーザー（企画者）が作ったものを、後でエンジニアが引き継ぐ」ケース
- lessons.md が仕様書・経緯説明として機能すればプロトタイプ→本開発の橋渡し
- B2B の現場で強力な価値になる

### P3 追加: scaffold の「第0歩目」フォロー
- PowerShell の Set-ExecutionPolicy や chmod +x は文系ユーザーの最大離脱ポイント
- 「外部ツール不要」でもOS標準ツールの設定変更は「高度な専門知識」に見える
- READMEで第0歩目を手厚くフォロー or 実行不要な代替手段を提供
