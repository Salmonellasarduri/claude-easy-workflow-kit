# /design — タスク設計・調査

## 目的

次の実装タスクを調査・設計し、受け入れ基準を設定してユーザーの承認を得る。

## 重要なゲート

**ユーザーに提示する前にレビューを通すこと（`/reviewing` の B パターン参照）。**

---

## 実行手順

### 1. コンテキスト確認

以下のファイルを読む（パスは `workflow.yaml` を参照）:
- タスクファイル — 現在のタスク
- レッスンファイル — 関連する過去の教訓
- `tasks/strategy_context.md` — **Read only if it exists.** Contains forging results from `/strategy_deep`. Incorporate acceptance criteria, resolved issues, and remaining risks into the plan

### 2. コードベース調査

変更対象が 2ファイル以上の場合、**Explore サブエージェント** に調査を委譲する。
詳細な手順は `.claude/skills/plan-researcher/SKILL.md` または `.claude/rules/research-guide.md` を参照。

サブエージェントを呼ぶ際は `.claude/rules/subagent-schema.md` の 4 項目 schema（結論 / 根拠 / 未確定点 / 参照先）を末尾に添えること。生出力をそのままメインに展開しない。

1ファイルで完結する場合は直接 Read で十分。

### Preferred Path（Codex がある場合）

- Explore の調査結果を Codex に送り、抜け漏れがないかレビューさせる
- プラン文面を `/reviewing` の B パターンで Codex にレビューさせる

### Fallback Path（Codex なし）

- Explore サブエージェントの調査結果のみで設計する
- プランの自己レビューチェックリスト:
  - [ ] 変更ファイルが漏れなく列挙されているか
  - [ ] 受け入れ基準が具体的で検証可能か
  - [ ] リスクが洗い出されているか
  - [ ] 既存パターンに従っているか

### 3. 実装プランを設計する

`schemas/handoff.md` の PlanArtifact 形式に従う:
- 変更するファイル一覧
- 実装ステップ（順序付き）
- 受け入れ基準
- リスク・注意点

### 4. レビュー結果を反映する

Preferred / Fallback いずれかのレビューで指摘された事項をプランに反映する。

### 5. 最終プランをユーザーに提示して承認を得る

### 6. ユーザーが承認したら

**自動続行（`workflow.auto_continue: true` — デフォルト）:**
- ユーザーが「はい」「OK」「進めて」等で承認した場合 → `/implement` を自動実行する

**手動モード（`auto_continue: false`）:**
- ここで停止し、ユーザーの指示を待つ

## 出力フォーマット

**「何が起きるか」を先に書き、技術詳細は後に置く。**

```
## 実装プラン: [タスク名]

### これで何が変わるか
[ユーザーの体験・プロダクトがどう変わるか — 平文 1〜3文]

### やること
1. [何をするか — 専門用語を最小限に]
2. ...
3. ...

### 確認ポイント
- [ ] [動作として分かる条件]
- [ ] ...

### 技術詳細（読み飛ばしてOK）
- `src/xxx.py` — [変更内容]
- リスク: ...
- レビュー結果: [lgtm / needs_fix]

この内容で進めていいですか？
```

**原則**:
- 「やること」は動作・結果で書く（「ファイルAにメソッドBを追加」ではなく「〜ができるようになる」）
- 技術詳細セクションは開発者向け — ユーザーが読み飛ばしても判断できる構成にする
