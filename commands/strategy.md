# /strategy — 戦略方向性の決定

## 目的

現在の状況を分析し、次の開発方向性の選択肢を提示してユーザーに判断を委ねる。

## 責任分担

- **Claude**: 分析・選択肢の提示・トレードオフの説明
- **ユーザー**: 最終的な方向性の決定

---

## 実行手順

### 1. 現状を把握する

以下のファイルを読む（パスは `workflow.yaml` を参照）:
- タスクファイル（デフォルト: `tasks/current.md`）— 現在のアクティブタスク
- ロードマップ（デフォルト: `docs/ROADMAP.md`）— フェーズ進捗
- 開発ログインデックス（デフォルト: `docs/DEVLOG.md`）— 直近のセッション記録
- レッスンファイル（デフォルト: `tasks/lessons.md`）— 関連する過去の教訓

### Preferred Path（外部ツールがある場合）

`workflow.yaml` の `tools` 設定に応じて追加の分析を行う:

- **gh: true** → `gh issue list` / `gh milestone list` で GitHub の実態を確認
- **gemini: true** → Gemini にドキュメントとコードの整合性チェックを委譲
- **codex: true** → 技術的リスクの分析を Codex に委譲

### Fallback Path（ツールなし）

- 上記ファイルの読み取りと自分自身の分析のみで判断する
- Explore サブエージェントでコードベースの実態を確認する

### 2. 現状を構造化して分析する

- 完了済みのもの
- 進行中のもの
- 技術的懸念点・ブロッカー

### 3. 選択肢を3案（A / B / C）で提示する

- 各案のメリット・デメリット・リスク
- Claude の推奨案を明示する（ただし決定はユーザーが行う）

### 4. ユーザーが選択したら

- タスクファイルに次のタスク群を書き込む（受け入れ基準付き）
- ロードマップの進捗を更新する（必要な場合のみ）
- **gh: true** の場合: GitHub に Milestone / Issues を作成する
- **ここで必ず停止する** — `/plan` への続行はユーザーが明示的に指示するまで待つ

---

```
### B: [タイトル]
...

### C: [タイトル]

どの方向性を選びますか？
```
