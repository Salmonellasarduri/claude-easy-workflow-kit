# /debug — デバッグ・動作検証ループ

## 目的

実装した変更に対して、テストデータを投入・検証し、クリーンアップするまでを体系的に回す。
**`/implement` と `/review` の内側から呼ばれる**（単独起動も可）。

---

## 実行手順

### 1. 検証対象を確認する

タスクファイル（`workflow.yaml` の `paths.tasks`）の受け入れ基準を読み、何を証明すれば完了かを明確にする。

### 2. テストを準備する

**原則: 専用テストスクリプトを作成する（`scripts/test_{name}.*`）**

受け入れ基準が1つ以上ある場合、専用スクリプトを作成してから実行する。
スクリプト末尾に `# PROOF:` コメントで「何を証明したか」を記録する:

```
# PROOF:
# - 受け入れ基準①: [実行コマンド] → [期待出力] → [実際の出力] → PASS/FAIL
# - 受け入れ基準②: ...
```

**例外（一行コマンドでよいケース）**: 単一の関数呼び出しで完結する軽微な修正のみ

### 3. 変更の性質に応じた追加パターン

**A: ドライランモード（外部 API・投稿系）**
- `dry_run=True` で副作用なしに実行パスを確認する

**B: 設定ファイルへの仮エントリ注入**
- 変更前に `.bak` を作成してから上書き
- 仮エントリは `_test_` プレフィックスで識別する

**C: データベースへの仮レコード挿入**
- rowid / primary key を必ず記録する

### 4. 検証ループ

```
スクリプト実行
  ↓
実行・観察（ログ確認 / 出力確認 / 期待値との照合）
  ↓
問題あり → デバッグ思考フロー → 修正 → 繰り返す
問題なし → クリーンアップへ
```

### 5. デバッグ思考フロー

```
バグ / エラー発生
  ↓
1. ログ・エラーメッセージを全文確認する
2. エラーが起きる最小ステップを特定する
3. 根本原因を特定する（症状ではなく原因を直す）
4. 複雑なバグ → Codex に相談（tools.codex: true の場合）
5. 修正後 → 検証ループに戻る
```

### 6. クリーンアップ（必須）

投入したテストデータを必ず除去する:

- **設定ファイル**: `.bak` から元に戻す
- **データベース**: 記録した rowid / key で DELETE する

**確認**: 仮データの残滓がないか検索する

---

## 完了条件

- [ ] テストスクリプトを作成し実行した（軽微修正は一行コマンドで可）
- [ ] 各受け入れ基準が `# PROOF:` コメントで PASS と記録されている
- [ ] エラーログに警告・エラーが出ていない
- [ ] 投入したテストデータがすべて除去されている
- [ ] `.bak` ファイルが残っていない
