# /implement — 実装・デバッグループ

## 目的

承認済みのプランを実行し、動作確認が取れるまでデバッグループを回す。

## 前提条件

- `/design` が完了しユーザーに承認されていること
- タスクファイル（`workflow.yaml` の `paths.tasks`）に以下が揃っていること:
  - 変更ファイル一覧
  - 実装ステップ（順序付き）
  - 受け入れ基準

---

## 実行手順

1. タスクファイルの現在タスクから以下を確認する:
   - 変更ファイル（実装対象を限定する）
   - 受け入れ基準（完了判定に使う）

2. プロジェクトのコーディング規約に従って実装する（`.claude/rules/coding-core.md` を参照）。`.claude/rules/scope-guard.md` の 5 禁止項目（指示外 rename / 意味非変更の大量 diff / ついでリファクタ / テスト期待値の根拠なき書き換え / 既存実装の巻き戻し）に該当しないかを実装中に意識する。

3. 実装後に **`/debugging`** を実行する:
   - テストデータ投入 → 検証 → クリーンアップの全手順は `/debugging` に従う
   - 完了条件（受け入れ基準証明・テストデータ除去）を満たすまでループする

4. `/debugging` 完了後 → `/reviewing` を呼ぶ

5. `/reviewing` 完了後 → 完了サマリーを表示して `/save` を提案する

---

## 完了条件

- [ ] 受け入れ基準がすべて満たされている（`/debugging` で証明済み）
- [ ] テストデータがすべて除去されている
- [ ] エラーログに警告・エラーが出ていない
- [ ] `/reviewing` を通過した

---

## Step 5: 完了サマリーと /save 提案

`/reviewing` が verdict: `lgtm` を返したら、以下の形式でサマリーを表示する:

```
## 実装が終わりました

### 何が変わったか
[ユーザーの体験・プロダクトがどう変わったか — 平文 1〜3文]

### 確認できたこと
- [受け入れ基準1の結果] ✓
- [受け入れ基準2の結果] ✓

レビューも通りました。保存しますか？
```

ユーザーが「はい」「OK」等で承認したら `/save` を実行する。

**注意**: `/save` は VCS 操作を含むため、`auto_continue` の設定に関わらず必ずユーザーの明示的な承認を得てから実行する。
