# /plan — タスク設計・調査

## 目的

次の実装タスクを調査・設計し、受け入れ基準を設定してユーザーの承認を得る。

## 重要なゲート

**ユーザーに提示する前にレビューを通すこと（`/review` の B パターン参照）。**

---

## 実行手順

### 1. コンテキスト確認

以下のファイルを読む（パスは `workflow.yaml` を参照）:
- タスクファイル — 現在のタスク
- レッスンファイル — 関連する過去の教訓

### 2. コードベース調査

変更対象が 2ファイル以上の場合、**Explore サブエージェント**に調査を委譲する
（`rules/research-guide.md` の手順に従う）。

1ファイルで完結する場合は直接 Read で十分。

### Preferred Path（Codex がある場合）

- Explore の調査結果を Codex に送り、抜け漏れがないかレビューさせる
- プラン文面を `/review` の B パターンで Codex にレビューさせる

### Fallback Path（Codex なし）

- Explore サブエージェントの調査結果のみで設計する
- プランの自己レビューチェックリスト:
  - [ ] 変更ファイルが漏れなく列挙されているか
  - [ ] 受け入れ基準が具体的で検証可能か
  - [ ] リスクが洗い出されているか
  - [ ] 既存パターンに従っているか

### 3. 実装プランを設計する

`schemas/handoff.md` の PlanArtifact 形式に従う:
- 変更するファイル一覧
- 実装ステップ（順序付き）
- 受け入れ基準
- リスク・注意点

### 4. レビュー結果を反映する

Preferred / Fallback いずれかのレビューで指摘された事項をプランに反映する。

### 5. 最終プランをユーザーに提示して承認を得る

---

## 出力フォーマット

```
## 実装プラン: [タスク名]

### 変更ファイル
- `src/xxx.py` — [変更内容]
- `src/yyy.py` — [変更内容]

### 実装ステップ
1. ...
2. ...
3. ...

### 受け入れ基準
- [ ] ...
- [ ] ...

### リスク・注意点
- ...

### レビュー結果（要約）
- verdict: [lgtm / needs_fix]
- [指摘事項と対応]

承認していただければ実装を開始します。
```
